\chapter{Introduction}
\section{Project Rationale}
The project aims to develop a replacement for the existing navigation service \href{https://map.csu.ru}{map.csu.ru}. The primary driver for development is the need to implement modern 360-degree panoramic imaging technologies to enhance the visualization of the university environment. During the conceptual discussion, it was decided to discontinue support for the current codebase in favor of developing a fundamentally new application. This decision is based on the following factors:
\begin{enumerate}
	\item low cost-effectiveness of maintaining the previous version due to accumulated technical debt and architectural complexity;
	\item the necessity of implementing a modern technology stack to ensure scalability and performance.
\end{enumerate}

The project objective is to develop a multifunctional ecosystem that addresses key university digitalization challenges and improves the Quality of Life (QoL) for students and staff. The implementation roadmap includes:
\begin{itemize}
	\item development of a cross-platform interactive map editor with 3D imagery support, event and schedule integration, and shortest-path routing algorithms;
	\item integration of an ejabberd server and a web client to facilitate internal communication;
	\item development of data import tools (parsers) or manual schedule entry interfaces.
\end{itemize}
\chapter{Theoretical Framework}
\section{Analysis of Current Challenges}
Currently, the academic process is associated with several organizational complexities. These include the absence of a unified secure environment for real-time communication between students and faculty, the decentralization of university information resources, and the low efficiency of notification channels.

The software product under development is designed to aggregate the necessary services into a single interface, thereby increasing the transparency and accessibility of the educational environment.

\section{Core Development Principles}
The key priorities in system design are security, performance, portability, and maintainability. These requirements dictated the choice of the technology stack. Several languages, including Python, JavaScript, and C\#, were excluded from consideration as they did not meet the performance criteria or the specific architectural requirements of the project.

Furthermore, a significant objective for the development team is to gain hands-on experience with high-performance systems and to master industry-relevant skills.

\section{Panoramic Content Visualization Methods}
To achieve a full sense of presence and indoor navigation, 360-degree panoramic imaging technologies are employed. The key task in the software processing of such content is the transformation of spherical data into a flat image and its subsequent re-projection within the user interface.

The primary standard here is the equirectangular projection, where the coordinates of each point on the sphere ($\theta, \phi$) are mapped to the Cartesian coordinates of the image pixels ($x, y$).

Within the scope of this project, 360-degree map visualization is implemented using the specialized rendering library \texttt{three.js}, which maps the equirectangular texture onto the inner surface of a virtual sphere or cube. To optimize client-side performance, a segmented loading (tiling) technique is applied, avoiding excessive RAM consumption when handling ultra-high-resolution images (8K and above).

\section{Routing Fundamentals}
The system is based on a mathematical model of a weighted directed graph (where each edge has a value), constructed according to the principle of Poincaré duality. In this model, 3D indoor volumes (rooms, corridors) are transformed into nodes (0D), while the connecting surfaces (doors) become edges (1D). The integration of the IndoorGML standard (describing 3D topology and XYZ coordinates) allows for dynamic network filtering via the \texttt{pgRouting} extension. This enables the calculation of the shortest path using Dijkstra's algorithm on an up-to-date subgraph that accounts for the physical connectivity of floors in real-time.

\section{Distributed Communication Protocols}
To establish a unified interaction environment for educational process participants, the XMPP (Extensible Messaging and Presence Protocol) was selected. Unlike proprietary solutions, XMPP is based on open XML standards and provides high extensibility through XEP (XMPP Extension Protocols) specifications.

The choice of the \texttt{ejabberd} server is driven by its fault tolerance, robust API, and ability to handle a large number of concurrent connections, which is critical for university-scale operations. In the system architecture, \texttt{ejabberd} serves as a module providing:
\begin{itemize}
	\item real-time instant messaging;
	\item isolation of personal social networks from the learning process;
	\item multi-user conference management for organizing study groups;
	\item user notification services.
\end{itemize}
The protocol integration ensures a communication channel decoupled from personal data and eliminates dependency on a specific client, while maintaining traffic and message encryption and excluding external infrastructure.

\section{Information Security}
Data protection is addressed at all stages of the Software Development Life Cycle (SDLC). The choice of the Go programming language is justified by its memory safety mechanisms. To minimize attack vectors, the following solutions are implemented:
\begin{itemize}
	\item use of ORM to prevent SQL injections;
	\item application of Anubis for protection against automated attacks (bots);
	\item implementation of specialized Middleware;
	\item use of JWT for user authorization;
	\item deployment in "distroless" containers to reduce the attack surface;
	\item integration of static analyzers and Quality Gates into the CI/CD process.
\end{itemize}
This approach implements the "Shift Left Security" concept, allowing for the identification and remediation of vulnerabilities at the early stages of development.

The designed system architecture is dictated by the specifics of the target infrastructure — deployment within an isolated virtual machine (VM) with limited computing resources (1 vCPU, 1 GB RAM). These conditions impose several constraints on the use of orchestration tools and necessitate a transition to a hybrid software development model.

\subsection{Infrastructure Characteristics and Design Principles}
The choice of a \texttt{docker-compose} based deployment environment limits horizontal scaling capabilities in the short term; however, it ensures high autonomy in system resource management. The core architectural principles include:
\begin{itemize}
	\item  {Service Consolidation:} Minimizing the number of containers to reduce virtualization overhead under resource-constrained conditions.
	\item  {Hybrid Monolith:} The core business logic is implemented within a single service (Backend), which simplifies inter-node communication while maintaining modularity for connecting external components (parsers and bots).
	\item  {Stateless Approach:} The backend is designed without session state persistence, ensuring the system's potential readiness for migration to a clustered environment (e.g., Kubernetes) for future scaling.
\end{itemize}

\subsection{Technology Stack Components}

{1. Networking and Security (NGINX, Anubis)} \\
To handle traffic routing and ensure information security, a combination of the NGINX web server and the Anubis protection system will be utilized. This solution is critical for preventing system performance degradation caused by unauthorized traffic spikes (automated attacks) under limited VM resource conditions.

	{2. Storage and Geospatial Analysis Layer (PostgreSQL, PostGIS, pgRouting)} \\
PostgreSQL has been selected as the primary DBMS. Its  {PostGIS} extension provides efficient storage and processing of spatial data. To implement navigation logic, the  {pgRouting} extension will be integrated, offering tools to execute shortest-path algorithms directly on the database side. This allows for offloading the main service by delegating complex graph computations to the optimized DBMS engine.

	{3. Communication Subsystem (Ejabberd and XMPP Client)} \\
The \texttt{ejabberd} server will be used to facilitate user interaction. A key feature of this integration is the delegation of user authentication to the main backend. The use of the OAuth protocol enables a Single Sign-On (SSO) concept, eliminating the need to duplicate and synchronize credentials across multiple data stores. The selection of a specific XMPP web client is currently under investigation.

	{4. Object Storage and Caching (MinIO, Redis)} \\
Static content (map tiles, 3D panoramas) will be stored in the S3-compatible \texttt{MinIO} object storage. \texttt{Redis} will be employed for task queue management and real-time data caching, significantly improving system response times.

	{5. External Integration Services (Parsers and Bots)} \\
Data collection modules (schedule parsers) are implemented in Python to ensure flexibility in processing structured information. For chat-bot development, priority is given to the Go language to minimize resource consumption and ensure native compatibility with the core backend architecture; however, Python may be used when specific protocol support is required.

\subsection{Component Initialization and Deployment Process}
The application startup is a strictly deterministic sequence of resource and service initialization stages. The deployment algorithm includes the following steps:

\begin{enumerate}
	\item  {Configuration:} Using the \texttt{Viper} library to load environment parameters from environment variables and configuration files. This allows for flexible adjustment of settings (ports, database addresses, access keys) without rebuilding the binary file.
	\item  {Database State Management:} Establishing a connection to PostgreSQL and executing automatic migrations using \texttt{golang-migrate}. This guarantees data schema consistency, including the \texttt{PostGIS} and \texttt{pgRouting} extensions.
	\item  {Caching and Queue Initialization:} Creating clients for \texttt{Redis} and \texttt{Asynq}, which are required for handling background tasks and transient data.
	\item  {Dependency Injection (DI):} Implementing the IoC (Inversion of Control) pattern. All objects (repositories, services, handlers) receive their dependencies explicitly through constructors. This approach eliminates hidden dependencies within components and facilitates unit testing.
	\item  {Transport Layer Startup:} Initializing HTTP server routes, the WebSocket hub, and activating background workers.
\end{enumerate}

\subsection{Transport Layer and Request Processing}
The \texttt{Gin} framework was selected as the primary HTTP framework, characterized by high performance and a robust \textit{middleware} ecosystem. The lifecycle of an incoming request includes:
\begin{itemize}
	\item Passing through a chain of \textit{middleware} (authentication, logging, attack protection).
	\item Routing to the corresponding handler.
	\item Invoking business logic through the service layer and returning a structured response.
\end{itemize}

To ensure interactivity and real-time event processing (e.g., during map editing or receiving notifications), a  {WebSocket Hub} subsystem has been implemented. It manages active connections and broadcasts updates between clients, minimizing latency compared to conventional HTTP requests.

\subsection{Abstraction Layers: Services and Repositories}
The application architecture is divided into logical levels to isolate business rules from infrastructure implementation details:

\begin{enumerate}
	\item  {Service Layer:} Encapsulates the core business logic of the project. Services operate through repository interfaces and external clients (\texttt{Redis}, \texttt{MinIO}), remaining independent of the specific storage implementation. This layer also handles interaction with  {pgRouting} to calculate optimal paths based on the provided coordinates.
	\item  {Data Access Layer (DAL):} Implemented via the "Repository" pattern. Repositories hide the details of SQL queries and the specifics of the \texttt{GORM} ORM.
	\item  {Interface Abstraction:} The use of interfaces allows for easy replacement of real database connections with \textit{mock} objects during testing, which is critical for ensuring code stability.
\end{enumerate}

\subsection{Background Tasks and Asynchronous Processing}
The \texttt{Asynq} library is used to perform resource-intensive operations, such as schedule parsing or bulk geospatial data processing. Tasks are placed into a \texttt{Redis} queue and processed by dedicated workers. This allows the main HTTP server to remain responsive by not blocking the execution flow during heavy computations.

\section{DevOps Methodology and Information Security}
A modern Continuous Integration and Continuous Delivery (CI/CD) cycle centered on the "Shift Left Security" concept has been implemented within the project. The primary objective of this approach is to identify defects and vulnerabilities at the earliest possible stages of development.

\subsection{Continuous Integration (CI) Pipeline and Quality Assurance}
The code build and verification process will be automated using the  {Jenkins} automation server. The lifecycle of changes in the codebase is strictly regulated and includes the following stages:

\begin{enumerate}
	\item  {Automated Testing:} Every \textit{PR} (Pull Request) triggers a suite of unit and integration tests. Passing all tests is a mandatory prerequisite for further code analysis.
	\item  {Static Analysis and Quality Gates:} The source code undergoes deep scanning in the  {SonarQube} system. The check includes searching for potential vulnerabilities, assessing technical debt, and analyzing code coverage. Branch merging is permitted only upon successfully passing the established \textit{Quality Gates}.
	\item  {Mandatory Code Review:} The branching policy prohibits direct pushes to the \texttt{main} branch. Any change requires review and formal approval from at least one other member of the development team.
\end{enumerate}

\subsection{Containerization and Runtime Security}
To minimize attack vectors during the software product's operational phase, containerization technology utilizing specialized images will be employed.

\begin{itemize}
	\item  {Distroless Containers:} Unlike standard base images (e.g., those based on Debian or Alpine), \textit{distroless} images contain exclusively the application and its runtime dependencies. They lack package managers, shells, and standard OS utilities. This significantly reduces the "attack surface" and eliminates the possibility of an attacker executing arbitrary scripts within the container.
	\item  {Multi-stage Builds:} The Go application is compiled in an intermediate container, after which the resulting binary is copied into a minimalist final image.
\end{itemize}

\subsection{Monitoring and Network Perimeter Protection}
The infrastructure layer is supplemented with proactive defense tools:
\begin{itemize}
	\item  {Middleware Filtering:} Implementation of an intermediary request-processing layer to validate incoming data and prevent common attacks (SQL injections, XSS).
	\item  {Anubis:} Utilizing the Anubis system in conjunction with NGINX allows for effective blocking of automated malicious traffic and botnets, maintaining VM resource availability for legitimate users.
\end{itemize}

This suite of measures ensures that only verified, tested, and secured code that meets established security standards reaches the runtime environment.

\chapter*{Conclusion}

In the course of this work, the architecture of a multifunctional university navigation ecosystem was designed. The developed solution will not only replace outdated visualization tools but also establish a foundation for a unified digital space within the university.

The key achievements of the project include:
\begin{itemize}
	\item  {Technological Modernization:} The transition to the Go language and a microservice-oriented approach ensured high system performance even under constrained server resources.
	\item  {Navigation Innovation:} The implementation of 360-degree panoramas in conjunction with the \texttt{pgRouting} mathematical framework and graph-based models allowed for the creation of an intuitive and accurate indoor routing tool.
	\item  {Security and Reliability:} Applying the "Shift Left Security" concept, utilizing \textit{distroless} containers, and maintaining a rigorous CI/CD pipeline guarantee the protection of user data and system resilience against external threats.
	\item  {Scalability:} The selected technology stack and the \textit{stateless} backend architecture allow for seamless project scaling as load increases or functionality expands.
\end{itemize}

Once the software product is developed, it will comply with modern development standards. Upon deployment within the university infrastructure, it will provide a significant improvement in the quality of interaction between students, faculty, and the university's digital environment.
